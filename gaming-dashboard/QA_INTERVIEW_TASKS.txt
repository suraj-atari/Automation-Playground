2. Test Case Prioritization
   - Which test cases should be automated first?
   - What would you leave for manual testing?
=================================================================================================================================================
Test Case Prioritization
1. Smoke and Regression Tests(Login/logout flow, Navigation between dashboard sections, Validation of total revenue, total games, and platform counts, Filters (e.g., platform dropdown), top 5 games verification)
2. Critical UI Validations (Presence and visibility of major components (cards, tables, graphs), Validation of table data (e.g., date formats, revenue format, currency symbol), Alignment and responsiveness of key widgets)
3. Data-driven Validations (Comparing displayed values with back-end data (CSV/JSON input), Verifying calculated totals and platform-wise revenue splits)

Keep for manual testing
1. Visual/UX assessments (color contrast, look & feel, alignment issues)
2. Ad-hoc exploratory testing to uncover edge cases
3. Newly developed or unstable features (until the functionality stabilizes)
4. Cross-browser visual differences that often need human judgment

3. Test Data Management
   - How would you handle the CSV and JSON test data?
   - Strategy for test data creation and maintenance
=================================================================================================================================================
Handling the CSV and JSON test data
I would take a data-driven testing approach so that test logic stays separate from the data. 
Keep all CSV and JSON files in a dedicated /test-data folder in the automation repo
1. For Json I will use ObjectMapper, is part of the Jackson library in Java. It’s used to read/write JSON data.
File file = new File("testdata/top-games.json"); points to the JSON file that contains test data. (Same way I have used in this project while automationg)
2. For CSV I will use FileReader -> Reads the CSV file from the specified path --> CSVParser (from Apache Commons CSV library) → Parses the CSV content into a structured format (rows & columns) that you can easily iterate over in tests (Same way I have used in this project while automationg)


Strategy for test data creation and maintenance
1. Prepare sample CSV files for Valid game names, platforms, revenues, revenue and dates
2. Use JSON files for Top games list, Platform configurations or expected totals\
3. Cover Different Test Scenarios (Include happy-path data (normal, expected records), Include edge-case data(Zero or negative revenue, Missing Dates or revenues, Extremely long game names)
4. Use separate data files for Dev, QA, and Staging environments if the dataset differs.
5. Use configuration files (e.g., config.json or environment variables) to switch between them without changing the test code.

Test Data Maintenance
1. Keep all test data in a dedicated folder like /test-data within the repository.
2. Version-control the data files alongside the code so changes are tracked.
3. Implement a simple utility to read data from CSV/JSON and pass it into tests dynamically. This way, any change in the data doesn’t require code changes in test cases.
4. Minimize Hard-Coded Data, making the locators more generic and robust, makes it easier to add new scenarios or modify existing ones without touching the automation code.

Deliverable: Test automation strategy and High-Level Test Automation Framework Proposal

A Hybrid Test Automation Framework using Selenium WebDriver + TestNG, built with Page Object Model (POM) and Data-Driven Testing, to ensure scalability, maintainability, and easy CI/CD integration.

Tool Selection - Selenium WebDriver – For browser automation, RestAssured for API Automation
TestNG – For test execution, grouping, prioritization, and parallel runs
Maven – For build and dependency management
POM (Page Object Model) – To separate page locators and actions for better reusability
Data-Driven Testing – Using CSV/JSON/Properties for test input and environment configs
Utility Layer – Common helper classes (e.g., data readers, wait handlers, API utilities)
Reporting – Using Allure Reports or Extent Reports for detailed insights
CI/CD Integration – Using Jenkins/GitHub Actions for automated execution in pipelines
